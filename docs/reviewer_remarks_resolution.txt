ENTRY 1

Reviewer question

Regarding Lemma 2.4, would it be better to state the actual conditions for
statistical hiding, rather than sketching the proof?

Current paper context

- The relevant text is Lemma `lem:ajtai-hiding` in
  `sections/02_preliminaries.tex`.
- The current statement says only that if the randomness dimension satisfies
  the leftover-hash entropy condition of Attema-Cramer-Kohl Lemma 1, then the
  commitment is statistically hiding.
- The current formulation does not spell out the entropy inequality in the
  notation of this paper, and therefore forces the reader to reconstruct the
  condition from the citation.

Why the issue is real

- The reviewer is correct that the proof sketch is not the useful part here.
  What matters for a reader is the exact parameter condition and the resulting
  hiding level.
- Attema-Cramer-Kohl Lemma 1 gives a direct leftover-hash condition for the
  bounded Ajtai commitment. In their notation, statistical hiding follows once
  the randomness block has enough min-entropy to statistically cover the output
  space.
- In our notation, each ring element contributes N coefficient slots. If the
  randomness block has dimension `ell_r` and coefficient alphabet
  `[-B_r, B_r]`, then the randomness min-entropy is
  `N * ell_r * log(2 B_r + 1)`.
- The commitment output lives in `R_q^{ell_Com}`, so the target output
  entropy is `N * ell_Com * log q`.
- Therefore the concrete hiding condition should be stated explicitly as

  `N * ell_r * log(2 B_r + 1) >= N * ell_Com * log q + 2 lambda`.

- In the current paper the commitment randomness uses the same coefficient
  bound as the message, namely `BoundMsg`, so for the present draft one may set
  `B_r = BoundMsg`.
- This is a pure exposition problem, not a conceptual proof problem.

Recommended answer

We should accept the reviewer's suggestion. The paper should state the hiding
condition directly and stop there. The correct answer is:

- the commitment is statistically hiding with statistical parameter `lambda`
  once the randomness block satisfies

  `N * ell_r * log(2 BoundMsg + 1) >= N * ell_Com * log q + 2 lambda`,

  or equivalently the same inequality with `B_r` if we later separate the
  randomness bound from the message bound;
- under that condition, the commitment distribution is within statistical
  distance at most `2^{-lambda}` of a message-independent distribution;
- the proof itself should be omitted and delegated to Attema-Cramer-Kohl
  Lemma 1.

Planned paper change

1. Rewrite Lemma `lem:ajtai-hiding` in `sections/02_preliminaries.tex` to state
   the explicit entropy inequality in the paper's notation.
2. State the resulting statistical parameter directly, rather than referencing
   a generic "leftover-hash entropy condition."
3. Remove the current proof-style narration and replace it with a short citation
   sentence to Attema-Cramer-Kohl Lemma 1.
4. If later revisions split message and randomness bounds, replace `BoundMsg`
   in the statement by a dedicated randomness bound `B_r`.

ENTRY 2

Reviewer question

For the cleared-denominator notation, does the stated identity really imply the
intended quotient relation? If `T`, `B_3 - r_1`, and
`B_0 + B_1(M_1 + M_2) + B_2 R_0` all vanish in one subfield, then the cleared
equation holds, but the quotient is still undefined there.

Current paper context

- The relevant text is Definition `def:rational-hash` and equation
  `eq:hash-cleared` in `sections/02_preliminaries.tex`.
- The current wording says:
  "Whenever `t = h_{m,(r_0,r_1)}(B)` is defined, it is equivalently
  characterized by the cleared-denominator identity ..."
- The same cleared relation is then reused in the NIZK-facing discussion of the
  pre-sign and post-sign statements.
- On the implementation side, `vSIS-HASH/vSIS-BBS.go` computes the quotient by
  explicitly inverting the denominator in the NTT domain and returns an error
  if any slot is non-invertible. `credential/helpers.go` and `issuance/flow.go`
  inherit that behavior.
- By contrast, `PIOP/credential_constraints.go` currently builds the proof-side
  residual only for the cleared identity. It does not independently certify
  denominator admissibility.

Why the issue is real

- The reviewer is correct. Over `R_q`, the cleared identity is necessary for a
  defined quotient but not sufficient.
- The reason is that `R_q` decomposes into CRT slots, and non-invertible
  elements are zero divisors. If the denominator vanishes in some slot, then
  the cleared product can also vanish in that slot without making the quotient
  meaningful there.
- Therefore the present wording "equivalently characterized" is too strong as
  written.
- This is not only an exposition problem. It exposes a possible semantic gap
  between:
  - the abstract hash relation, which is defined only when `B_3 - r_1` is
    slotwise invertible, and
  - the current proof surface, which appears to replay only the cleared
    identity.
- As long as admissibility is not separately enforced in the proof relation, we
  must not claim that the NIZK proves the defined quotient relation itself.

Recommended answer

The right answer is:

- define the hash relation as the conjunction of
  1. slotwise invertibility of `B_3 - r_1`, and
  2. the cleared identity
     `(B_3 - r_1) * t - (B_0 + B_1 m + B_2 r_0) = 0`;
- only under the admissibility condition does the cleared identity become
  equivalent to the quotient equation;
- the paper must stop using the cleared identity alone as an equivalence claim.

Internally, we should treat this as the highest-priority issue in the current
review batch. The reviewer is pointing at a real mathematical distinction.

Planned paper change

1. In Definition `def:rational-hash`, replace the current equivalence sentence
   with a two-part definition:
   - admissibility: `B_3 - r_1` is invertible in every slot;
   - relation: the cleared identity holds.
2. Reword all later references so they say one of the following, depending on
   context:
   - "under the admissibility condition, the quotient is equivalent to the
     cleared identity", or
   - "the proof replays the cleared consistency relation for a public target
     already computed from an admissible denominator."
3. Audit `sections/03_blind_signature.tex`, `sections/04_arc_construction.tex`,
   and `sections/05_smallwood_model.tex` for any sentence that implicitly
   conflates admissibility with the cleared identity.
4. Outside the paper text, resolve whether the proof layer itself must grow an
   explicit denominator-admissibility constraint, or whether the final paper
   should instead present the NIZK more narrowly as proving consistency with a
   public target that was already generated only in the admissible case.
5. Do not send any external reviewer response that claims "the identity is
   equivalent as written." The correct response is that the reviewer is right
   and the side condition must be stated explicitly.

ENTRY 3

Reviewer question

For "Definition 2.10 and the rest of Section 2.3":

1. What are the formal security properties of the signature layer?
   Is it only sEUF-RMA? If so, what is the exact security experiment?
2. What assumptions do the trapdoor hash instantiation and trapdoor sampler
   rely on, and can the paper summarize the reductions cleanly in one lemma
   instead of scattering them through remarks?

Current paper context

- The relevant material starts in `sections/02_preliminaries.tex` at
  `def:vsis-template`, `def:dklw-assumptions`, and
  `rem:vsis-unforgeability`.
- It continues in the blind-signature syntax and game definitions
  `def:bs-syntax`, `def:bs-correctness`, `def:bs-blindness`,
  `def:bs-euf-cma`, and then in the security discussion of
  `sections/03_blind_signature.tex`.
- ARC-level consequences are then sketched in
  `sections/04_arc_construction.tex`.
- The current draft mixes three different layers:
  1. the non-blind vSIS/rational-hash signature template;
  2. the interactive blind-signature issuance protocol;
  3. the ARC showing protocol built on top of the blind signature and the PRF.
- The current DKLW remark justifies the non-blind layer, but nearby prose in
  Sections 3 and 4 informally leans on stronger blind-signature and ARC claims.

Why the issue is real

- The reviewer is right that the current exposition does not cleanly tell the
  reader which security claims are theorem-backed today and which are still
  only a roadmap.
- The DKLW theorems checked for this note support the following non-blind
  statements:
  - DKLW Theorem 4: correctness of the non-blind `Sigma_vSIS` template under
    admissible `(TrapGen, SampPre)` parameters, negligible inadmissibility
    `delta`, and the shortness condition on `beta`;
  - DKLW Theorem 5: sEUF-RMA security of the non-blind signature layer from the
    `s-$Hint-vSIS` assumption with parameters
    `(R, n, k, q, beta, F, G, H, P, Q)`;
  - DKLW Theorem 7: equivalence between the non-blind sEUF-RMA experiment and
    `GenISIS_f` for `f(B, m, chi) = Eval(B, m, chi)`;
  - DKLW Theorem 8 and Corollary 1: the link back and forth between
    `GenISIS_f` and `s-$Hint-vSIS` under the stated parameter translation and
    predicate conditions.
- By contrast, the current paper does not yet provide a completed reduction for
  one-more unforgeability or blindness of the blind-signature issuance
  protocol.
- The ARC layer is even further removed: its privacy and soundness discussion
  currently depends on blind-signature security, PRF security, and post-sign
  NIZK properties, but is still presented only at the sketch level.
- So the real issue is not that the paper cites the wrong source. The issue is
  that it does not separate theorem-backed non-blind results from blind and
  ARC-level roadmap claims.

Recommended answer

We should answer this by separating the security story into three layers and
being explicit about which one is already reduced.

Layer 1: non-blind vSIS/rational-hash signature layer

- This is the only layer for which the current draft has a concrete reduction
  basis.
- The paper should locally define, or explicitly import from DKLW, the
  non-blind security experiment for the signature template.
- The precise claim we can currently make is:
  - correctness under admissible `(TrapGen, SampPre)` parameters, negligible
    inadmissibility `delta`, and the DKLW shortness condition on `beta`;
  - sEUF-RMA security from `s-$Hint-vSIS` with parameters
    `(R, n, k, q, beta, F, G, H, P, Q)`;
  - equivalently, via DKLW Theorems 7 and 8 plus Corollary 1, the same
    non-blind layer can be phrased through `GenISIS_f`.

Layer 2: blind-signature issuance layer

- This layer has syntax and target security notions in the current draft, but
  not a finished reduction.
- The right description today is a roadmap, not a theorem:
  one-more unforgeability and blindness would need to combine
  extractability/knowledge soundness of the pre-sign proof with commitment
  hiding, zero knowledge, and an interactive one-more style assumption, likely
  closer to `IntGenISIS_f` or another blind-signature specific lifting of the
  non-blind layer.
- Therefore the paper should not present one-more unforgeability or blindness
  as already reduced.

Layer 3: ARC presentation layer

- This layer should be presented as building on the blind-signature layer plus
  PRF security and post-sign NIZK properties.
- Presentation unlinkability can remain as a formal game definition, but the
  proof discussion should stay clearly conditional and sketch-level until the
  blind-signature reduction is in place.

Planned paper change

1. In `sections/02_preliminaries.tex`, add a local non-blind signature security
   experiment or explicitly import DKLW's experiment near `def:vsis-template`,
   so later claims can point to a named experiment rather than to an informal
   remark.
2. Replace `rem:vsis-unforgeability` by a short theorem- or lemma-level summary
   that states exactly:
   - non-blind correctness under the DKLW parameter conditions;
   - non-blind sEUF-RMA under `s-$Hint-vSIS`;
   - the connection to `GenISIS_f` via Theorems 7 and 8 and Corollary 1.
3. In `sections/03_blind_signature.tex`, keep the blind-signature correctness
   proposition, but rewrite the current "Security roadmap" so it is explicitly
   labeled as unfinished reduction work rather than as a near-theorem.
4. In `sections/04_arc_construction.tex`, keep ARC correctness and
   unlinkability/soundness as conditional discussion only, and make sure no
   sentence suggests that one-more unforgeability or ARC privacy is already
   formally reduced in this draft.
5. If the paper later wants one compact summary statement, place it in
   Section 2 after the vSIS template and before the blind-signature syntax, so
   the reader sees the assumption layer before the protocol layer.

the paper must separate exact parameter conditions, exact semantic side
conditions, and exact theorem-backed security claims. The current draft is
closest to being complete on the non-blind signature layer, noticeably weaker
on the blind-signature layer, and still only sketch-level on the ARC layer.
